# GitHub Pages
## 01 サイトを公開する手順を確認

### GitHub Pages について
- できること
  * 無料で使える
  * HTML/CSS/JavaScript で作ったサイトを公開できる

- できないこと
  + PHP や Python、WordPress などは使えない
  + 容量や転送量に制限がある
  + 個人プランではアクセス制限できない
  + 商用利用できない

### サイト公開の流れ
1. サイトの制作
1. リポジトリの作成
1. サイトのアップロード
1. 公開設定
1. サイトの運用

## 02 公開するサイトを制作

## 03 GitHubでリポジトリを作成

### リポジトリ作成方法
1. https://github.com/new にアクセス
1. General の Repository name にリポジトリ名を入力
1. Choose visibility を選択
   - Public・・・アップロードされたコードが全世界に公開される
   - Private
1. Create repository をクリック

## 04 公開するサイトをアップロード

### サイトのアップロード方法
1. `uploading an existing file` をクリック
1. ファイルをドラッグ＆ドロップ

## 05 サイトの公開設定

### サイトの公開方法
1. リポジトリ内で `Settings` タブをクリック
1. サイドバーから `Pages` をクリック
1. Branch の項目を `main`、`/(root)` として `Save`
   - サイトを公開するための処理が実行
   - 進捗は `Actions` タブで確認
     + `pages build and deployment` をクリック
     + `Status` が `Success` で実行完了
1. リポジトリ内で `Settings` タブに戻り、`Pages` をクリック
1. `Visit site` をクリックして、別タブでサイトを確認

## 06 サイトの更新

### サイトの更新方法
1. サイトを修正
1. `Add file` の中の `Upload files` をクリック
1. 修正したファイルをドラッグ＆ドロップ

## 07 サイトの公開を停止

### サイトの公開停止方法
1. リポジトリ内で `Settings` タブに戻り、`Pages` をクリック
1. `Unpublish site` をクリック

### サイトの再度公開方法
1. `Actions` タブに戻り、一番上の `pages build and deployment` をクリック
1. 右上にある `Re-run all jobs` をクリック
   - 処理を再度実行する

### リポジトリごと削除する方法
1. `Settings` のタブをクリック
1. 一番下の `Delete this repository` をクリック

## 17 `git reset` の挙動
### 復習
#### 直近のコミットを上書きする手順
---------------------------------------
1. 直近のコミットを上書きする
- メッセージを追加する場合
  + `git commit --amend -m "メッセージ"`

- コミットメッセージはそのままでいい場合
  + `git commit --amend --no-edit`
---------------------------------------

#### 過去に戻る手順
---------------------------------------
1. | `git log --oneline` | コミットの一覧を簡潔に表示 |
1. |`git checkout [戻りたい hash]` | 任意のコミットの時点に戻る |
   - メッセージに出てきたコマンドを使用すると元に戻れる
1. | `git switch -` | 元に戻す |
---------------------------------------

- | `git reset [option] [hash]` | 任意のコミット時点にまで戻ることができる |
  + 指定したコミットから先が基本的には失われてしまう点に注意
  + 既にコミット履歴を他の人と共有していた場合、`git reset` を使ってその履歴を消してしまうと混乱のもとになる
  + `git reset` を使うときは、まだ他の人と共有していない履歴に対してのみ使う


#### リポジトリの状況
####  05 のテキストを追加したので git add でステージングに追加したけれども、まだコミットは作っていない、という状況
- 05 を追加
- 04 を追加
- 03 を追加
- 02 を追加
- 01 を追加

### [option] の種類
  - `git reset --hard`
    + 02 のコミットを指定した場合、そこから先のコミット内容が失われて、02 時点での内容にすべてが上書きされるので、ワーキングディレクトリとステージングエリアの02 より先の変更内容は破棄される

  - `git reset --soft`
    + 02 のコミットを指定した場合、コミットは失われるが、それ以降の変更はワーキングディレクトリーにもステージングにもキープしてくれる

  - `git reset --mixed`
    + 02 のコミットを指定した場合、変更内容もキープされるが、変更内容はワーキングディレクトリーにのみにキープされる
    + `git reset` のデフォルトのオプションは `--mixed`
    + `git reset --mixed [hash]` と `git reset [hash]` は同じ

## 18 複数のコミットを一つにまとめる
### 複数のコミットを一つにまとめる手順
---------------------------------------
1. git reset を使って、コミット履歴を戻す
1. 履歴は消える
1. リセットする際には、それ以降の変更内容をワーキングディレクトリに戻すようにする
1. 変更内容を新しく一つのコミットとして作り直せば、結果としてこれらのコミットを一つにまとめることができる
---------------------------------------

## 19 `git reset` でコミットをまとめる
- `git reset --mixed 8aaaac3`
- `git commit -am "いろいろまとめを追加"`

# ブランチ編
## 01 ブランチについて

### [用語] ブランチ
---------------------------------------
- 概要：前のコミットの情報を持っていて、その連なりのこと
- メモ：
  + ブランチは枝という意味だが、木の枝のように分岐させていくこともできて、そうすることで違うバージョンの開発を同時に進めることができる、という仕組みになっている
  + 最初に作られるブランチでは `main` という名前にするのが一般的
  + ブランチの使い方は開発の現場によって様々だが、`main` ブランチにはいつでも公開可能な安定したバージョンを保存しておいて、新機能はそこから派生するブランチで開発を始めて、その開発が完成したら `main` ブランチの方に統合していく、といった使い方をする
---------------------------------------

### [用語] マージ
---------------------------------------
- 概要：ブランチを統合すること
- メモ：
  + 新機能を開発中に既に公開している過去のバージョンに不具合が見つかって、すぐに対処する必要が出てきた場合、`main` ブランチでのみ開発を行っていると、変更を戻して必要な箇所を修正する必要があるので、新機能の開発が止まってしまう。また不具合が解決した後に、また元の箇所まで戻した時に、その影響で新機能の方が動かなくなってしまうことも考えられる。
---------------------------------------

#### `Git` では、今どちらのブランチで作業しているかを、現在のブランチの最新のコミットに `HEAD` というラベルを付けることで区別している

## 02 コミットを保存

1. | `git init` |フォルダを Git 用に初期化 |
  - 結果
    + Initialized empty Git repository ... と出ていれば OK
1. ファイルの作成
1. | `git add ファイル名` | ワーキングディレクトリからステージングにファイルを移す |
1. | `git commit -m "01を追加"` | リポジトリにコミットとして保存 |
1. | `git commit -am "02を追加"` | `add` してそのまま `commit` する |
1. | `git log --oneline` | コミットの一覧を簡潔に表示 |
  - 結果
    ```
    6ef6259 (HEAD -> main) 02を追加
    42465a3 01を追加
    ```
    + `HEAD` があるので、現在のブランチの最新のコミットは `main` ブランチのコミットになる

## 03 ブランチの操作
#### `main` ブランチはそのまま残しつつ、ブランチを分岐させて、別の開発を進めていきたい

- | `git branch` | ブランチの一覧を見る |

#### ブランチの操作
---------------------------------------
1. | `git branch add-03` | 新しくブランチを作成 |
  - ブランチに使える名前には細かいルールもあるが、分かりやすい単語を - でつないでおけば大体 OK
2. | `git branch` | ブランチの一覧を見る |
  - 結果
    ```
    add-03
    * main
    ```
    + 先頭の * は現在のブランチを指している
3. | `git switch add-03` | ブランチの切り替え |
4. | `git branch` | ブランチの一覧を見る |
  - 結果
    ```
    9236ea1 (HEAD -> add-03) 03を追加
    6ef6259 (main) 02を追加
    42465a3 01を追加
    ```

#### 今回 `main` はそのままにして、`add-03` ブランチでのみ、開発が進んでいるので、`main` ブランチに切り替えたら、`add-03`のコミットは見えなくなる

## 05 マージの挙動について

### [用語] マージ
---------------------------------------
- 概要：ブランチを統合すること
- メモ：
  + 新機能を開発中に既に公開している過去のバージョンに不具合が見つかって、すぐに対処する必要が出てきた場合、`main` ブランチでのみ開発を行っていると、変更を戻して必要な箇所を修正する必要があるので、新機能の開発が止まってしまう。また不具合が解決した後に、また元の箇所まで戻した時に、その影響で新機能の方が動かなくなってしまうことも考えられる。

   HEAD
   main
   *

   |/
   *
   *

1. | `git merge add-03 --no-ff` | マージコミットを作る |

   ```
   HEAD
   main
   * マージコミット
   |\
   | * add-03
   |/
   *
   *
   ```

   ---------------------------------------
   - 概要：マージのために作ったコミット

    | * 3dae75c (add-03) 03を追加
    |/
    * 6ef6259 02を追加
    * 42465a3 01を追加
    ```

#### コミットがない状態でマージする場合、デフォルトでは `Fast Forward` マージになるので、マージコミットを作るためには `--no-ff` オプションを付ける必要がある。しかし、分岐したあとで、両方のブランチにコミットがある場合、そのままマージするだけでマージコミットが作られるマージになる。

## 10 Fast-forwardではないマージを使用
7. | `git merge add-03 --no-edit` | ブランチをマージする（メッセージは自動生成） |

8. | `git branch -d add-03` | `add-03` ブランチを削除する |

   分岐したあとに、両方のブランチでコミットがある場合、デフォルトでマージコミットが作られるマージになる

         * 42465a3 01を追加

   2. | `git reset --hard [hash]` | コミット履歴を戻す |

2. readme.md を編集

- `main` と `add-03` で同じ箇所を変更しているため、マージしたあとにどちらの内容を選択していいか、Git が判断することができない

## 15 `git rebase` の挙動
### 開発の規模がある程度大きくなって、ブランチごとに担当者が割り当てられる場合

1. `add-03` のブランチで作業が完了
```
main
*
| * add-03
|/
*
|
*
```

2. コンフリクトが起こる
  - コンフリクトを解消するのは `main` 担当の人になってしまう

### [用語] リベース
---------------------------------------
- 概要：`add-03` のブランチの分岐元を、`main` のブランチの最新のコミットに付け替えること
- メモ：ブランチの分岐元、ベースを付け替えるという意味
---------------------------------------

```
main
*
|\
| * add-03
|/
*
|
*
```

#### リベースをすると、`main` の最新のコミットから分岐して開発したのと同じことになるので、その後に `main` とマージする時にコンフリクトが起こることがない

## 16 両方のブランチにコミットを保存
#### 【add-03 担当】
   1. | `git switch -c add-03` | 新しくブランチを作って、切替える |

   2. readme.md を編集

   3. | `git commit -am "03を追加"` | `add` してそのまま `commit` する |

#### 【main 担当】
   1. | `git switch main` | ブランチの切り替え |

   2. | `git commit -am "03を仮置き"` | `add` してそのまま `commit` する |

#### 【add-03 担当】
   4. | `git switch add-03` | ブランチの切り替え |

## 17 `git rebase` を使用
   5. | `git rebase main` | `add-03` の分岐元を `main` の先頭に持っていく |

   6. コンフリクトが起きる
   - エラー文
     ```
     hint: "git add/rm <conflicted_files>", then run "git rebase --continue".・・・解消した後に、git rebase --continue のコマンドを実行して

     hint: To abort and get back to the state before "git rebase", run "git rebase --abort".・・・abort オプションでリベースを中断する
     ```

   7. コンフリクトを解消する

   8. | `git commit -a --no-edit` | `add` してそのまま `commit` する（メッセージは自動生成） |


   9. エラー文記載のコマンド入力
      - `git rebase --continue`

#### 【main 担当】
   3. | `git switch main` | ブランチの切り替え |

   4. | `git merge add-03 --no-ff --no-edit` | ブランチをマージする（メッセージは自動生成） |

   5. | `git branch -d add-03` | `add-03` ブランチを削除する |

## 01 リモートリポジトリ

### [用語] リモートリポジトリ
---------------------------------------

---------------------------------------

### [用語] ローカルリポジトリ
---------------------------------------

## 02 GitHub でリモートリポジトリを作成
1. https://github.com/new にアクセス
1. General の Repository name にリポジトリ名を入力
1. Choose visibility を選択
   - Public・・・アップロードされたコードが全世界に公開される
   - Private
1. Create repository をクリック

## 03 ローカルリポジトリを設定
1. | `git init` |フォルダを Git 用に初期化 |
  - 結果
    + Initialized empty Git repository ... と出ていれば OK
1. ファイルの作成
1. | `git add ファイル名` | ワーキングディレクトリからステージングにファイルを移す |
1. | `git commit -m "01を追加"` | リポジトリにコミットとして保存 |
1. | `git commit -am "02を追加"` | `add` してそのまま `commit` する |
1. | `git log --oneline` | コミットの一覧を簡潔に表示 |
  - 結果
    ```
    8db3dcc (HEAD -> main) 02を追加
    8e061f9 01を追加
    ```
    + `HEAD` があるので、現在のブランチの最新のコミットは `main` ブランチのコミットになる

## 04 作業の流れを確認
0. GitHub にリモートリポジトリ、手元の PC にローカルリポジトリを設定した
1. | `git remote` | リモートリポジトリにアクセスしやすいように名前を付ける |
   - origin という名前を付けるのが一般的
2. | `git push` | ローカルリポジトリの内容をリモートリポジトリにアップロードして、両方のリポジトリを同期させる |
   - リモートリポジトリにアップロードする際には、ローカルリポジトリで「リモート追跡ブランチ」という特殊なブランチを一緒に作るのが一般的
3. リモートリポジトリで誰かが変更を加える
4. リモートリポジトリの内容をローカルリポジトリの方にも反映させる
   - Git ではローカルリポジトリの `main` ブランチに直接反映させない
   - 方法
     1. | `git fetch` | リモートブランチの内容をリモート追跡ブランチにダウンロード |
     2. リモート追跡ブランチを `main` ブランチにマージする
   - **注意**
     + `main` ブランチとリモート追跡ブランチをマージする際には、マージコミットを作ってしまうとリモートブランチと同じにならないため、マージコミットを作らない Fast Forward 形式でブランチのマージをすることになる

## 05 `git remote` コマンド
1. | `git remote` | リモートリポジトリにアクセスしやすいように名前を付ける |
   - GitHub のサイトにコマンドが記載されている
   - `git remote add origin https://github.com/atsu-3110/MyGitRemotes.git`
   - git のリモートリポジトリとして、URL にあるリモートリポジトリを origin という名前で追加して

2. | `git remote -v` | リモートリポジトリの設定を確認する方法 |
   - 結果
     ```
     origin  https://github.com/atsu-3110/MyGitRemotes.git (fetch)
     origin  https://github.com/atsu-3110/MyGitRemotes.git (push)
     ```

## 06 `git push` コマンド
1. | `git push` | ローカルリポジトリの内容をリモートリポジトリにアップロードして、両方のリポジトリを同期させる |
   - GitHub のサイトにコマンドが記載されている
   - `git push -u origin main`
   -  `origin` というリモートリポジトリに対して、ローカルブランチの `main` の内容をアップロード（`push`）する
   - | `-u` | 最初に一度だけ設定すれば OK で、これを付けておくことでリモート追跡ブランチを作ってくれたり、今後 `push` するときは `origin` や `main` を省略することができる |

   - エラー
     ```
     error: failed to push some refs to 'https://github.com/ユーザー名/MyGitRemotes.git'
     ```

   - 解決方法
     + 方法 1
       * Emails の Keep my email addresses private を off
     + 方法 2
       * `git config user.email "数字 + ユーザー名@users.noreply.github.com"`

## 09 `git fetch` コマンド
0. GitHub（リモートリポジトリ） で新しいコミットを作成
1. | `git branch -a` | リモート追跡ブランチも含めたブランチの一覧を見る |
   - | `-a` | all を意味する |
   - リモート追跡ブランチは、あくまでもリモートリポジトリの内容を取り込むためのもので、リモート追跡ブランチに切り替えたり、リモート追跡ブランチに直接コミットを加えることはできない
   - 結果
     ```
     * main
     remotes/origin/main
     ```

2. | `git log origin/main --oneline` | リモート追跡ブランチも含めたコミットの一覧を表示 |
   - `remotes` は省略可能
   - 結果
     ```
     12be1f2 (HEAD -> main, origin/main) 02を追加
     8e061f9 01を追加
     ```

3. | `git fetch` | リモートブランチの内容をリモート追跡ブランチにダウンロード |

4. | `git log origin/main --oneline` | リモート追跡ブランチも含めたコミットの一覧を表示 |
   - 結果
     ```
     527c511 (origin/main, origin/HEAD) GitHubで03を追加
     12be1f2 (HEAD -> main, origin/main) 02を追加
     8e061f9 01を追加
     ```

## 10 リポジトリの同期
5. | `git merge origin/main` | Fast Forward マージ |
   - **注意**
     + `main` ブランチとリモート追跡ブランチをマージする際には、マージコミットを作ってしまうとリモートブランチと同じにならないため、マージコミットを作らない Fast Forward 形式でブランチのマージをすることになる

## 11 プルリクエストについて
```

*

#### アプリケーションを開発するときは、`main` ブランチを担当する人がいて、新しい機能を追加する際には、ブランチを分けた上で、別の担当者がそちらを開発していく

#### ある時点で `add-03` の開発が完了して、`add-03` の変更を `main` にマージしてね、とお願いすることになるが、その際に `main` 担当の人は、完成したコードをレビューしてからマージするのが一般的
